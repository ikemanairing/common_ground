# Common Ground 구현 계획서 (Orchestration Plan)

## 목표
- 1~10 화면을 실제로 동작하는 모바일 웹앱(모바일 사용자 흐름, Mobile User Flow)으로 운영 가능 상태까지 확장한다.
- 기존 정적 HTML 자산을 보존하면서, 새 앱 구조(`app/`)와 서버 연동 구조(`supabase/`)를 동시에 만든다.

## 작업 원칙
- 6명의 서브에이전트가 파일 소유권(Ownership) 기준으로 병렬 구현한다.
- 에이전트끼리 같은 파일을 동시에 수정하지 않는다.
- 공용 계약(요청/응답 타입, API Contract) 먼저 고정하고 UI를 연결한다.

## 에이전트 분업
1. A1: 앱 뼈대(React/Vite/Router) 생성
2. A2: 단계 화면 1~5 컴포넌트 구현
3. A3: 단계 화면 6~10 + 짝표시(2인/3인) 컴포넌트 구현
4. A4: 공용 상태 저장소 + API 클라이언트 + 타입 계약 구현
5. A5: Supabase 마이그레이션 + Edge Function 스켈레톤 구현
6. A6: 테스트/문서/배포 스크립트 구현

## 산출물 구조
- `app/`: 프론트엔드 앱 코드
- `supabase/migrations/`: DB 스키마 SQL
- `supabase/functions/`: 서버리스 함수
- `docs/`: 운영/배포 문서
- `scripts/`: 개발 및 검증 스크립트

## 완료 기준
- `app/`에서 라우팅과 단계 화면이 모두 연결되어야 한다.
- 공용 상태 저장(로컬 저장소, LocalStorage) 및 복구가 동작해야 한다.
- API 호출 인터페이스가 타입으로 고정되어야 한다.
- Supabase 스키마와 함수 스켈레톤이 준비되어야 한다.
- 실행/검증 방법이 문서화되어야 한다.

## 병합 순서

2. A1(뼈대)
3. A2, A3(UI 단계)
4. A5(서버)
5. A6(테스트/문서)
